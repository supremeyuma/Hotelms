# Reporting Architecture - Migration Resolution Summary

## Issue Encountered

When running `php artisan migrate`, the reporting layer migrations encountered several issues:

1. **SQLSTATE[42S21]: Duplicate column name 'created_at'**
   - Problem: `reporting_order_facts` and `reporting_laundry_facts` tables defined `created_at` manually AND used `timestamps()` helper
   - Impact: MySQL saw two `created_at` column definitions in the same table

2. **SQLSTATE[42000]: Duplicate key name**
   - Problem: `payment_method` column indexed both inline (`->index()`) and explicitly (`$table->index(['payment_method'])`)
   - Impact: MySQL tried to create duplicate indexes on same column

3. **SQLSTATE[42000]: Identifier name too long**
   - Problem: Unique constraint name auto-generated by Laravel `reporting_exceptions_reference_type_reference_id_exception_type_unique` exceeds MySQL's 64-character limit
   - Impact: Migration failed before constraint could be created

## Solutions Applied

### Fix 1: Remove Duplicate `created_at` Definitions

**File**: `2026_04_12_000006_create_reporting_order_facts_table.php`

Removed:
```php
$table->timestamp('created_at')->index();  // REMOVED
$table->timestamp('accepted_at')->nullable();
```

Reason: `$table->timestamps()` already creates `created_at` and `updated_at` columns

### Fix 2: Remove Duplicate `created_at` Definition

**File**: `2026_04_12_000007_create_reporting_laundry_facts_table.php`

Same fix as above - let `timestamps()` handle it.

### Fix 3: Remove Duplicate Index

**File**: `2026_04_12_000010_create_reporting_payment_facts_table.php`

Removed:
```php
$table->index(['payment_method']);  // REMOVED - already indexed inline
```

Reason: Column already indexed with `->index()` on the column definition

### Fix 4: Rename Unique Constraint

**File**: `2026_04_12_000011_create_reporting_exceptions_table.php`

Changed:
```php
// Before:
$table->unique(['reference_type', 'reference_id', 'exception_type']);

// After:
$table->unique(['reference_type', 'reference_id', 'exception_type'], 'ref_exception_unique');
```

Reason: Auto-generated name exceeded 64-character MySQL identifier limit

### Fix 5: Remove Duplicate Index

**File**: `2026_04_12_000011_create_reporting_exceptions_table.php`

Removed:
```php
$table->index(['detected_at']);  // REMOVED - already indexed inline
```

Reason: `detected_at` column already has `->index()` on definition

## Best Practices Applied

### 1. Single Source of Truth for Timestamps
- Use `$table->timestamps()` for `created_at` and `updated_at`
- Don't manually define these columns
- Add additional timestamp columns only when needed

### 2. Index Management
- Define inline indexes with `->index()` on column definitions
- Don't duplicate with explicit `$table->index()` calls
- Use explicit indexes only for multi-column indexes

### 3. Constraint Naming
- Always provide explicit names for long constraints
- Keep names under 64 characters for MySQL compatibility
- Use abbreviated names like `ref_exception_unique` instead of auto-generated names

### 4. Migration Testing
- Test migrations in development first
- Verify schema with database tools
- Check for orphaned tables if migrations fail partway

## Lessons Learned

| Issue | Lesson | Prevention |
|-------|--------|-----------|
| Duplicate created_at | Always use timestamps() helper | Don't manually define timestamp columns |
| Duplicate indexes | Single index definition per column | Use either inline or explicit, not both |
| Long constraint names | MySQL has 64-char identifier limit | Provide explicit short names for constraints |
| Failed migrations orphan tables | Partial migrations leave tables behind | Clean up with Schema::dropIfExists() after failure |
